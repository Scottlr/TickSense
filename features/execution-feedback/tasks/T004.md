### T004 - Perfect Tick Feedback From Opportunities

#### Objective
Emit `Tick Perfect!` feedback when an opportunity completes with confirmed zero tick loss. This task creates domain feedback only; rendering and sound are later.

#### Scope
- Add a feedback sink to the activity engine or services composition.
- Convert selected `OpportunityMarker` completions into `ExecutionFeedbackEvent` records.
- Use only terminal `COMPLETED` markers with explicit zero-latency/tick-loss context.
- Add tests proving no feedback is emitted for late, failed, expired, cancelled, or ambiguous opportunities.

#### Out of Scope
- Do not render text.
- Do not play sound.
- Do not infer misclicks.
- Do not add report fields.
- Do not treat every completed opportunity as perfect.

#### Blocked By
T001, T003

#### Why It Is Blocked
The feedback model and config toggles must exist before opportunity-derived feedback can be emitted safely.

#### Implementation Details
Create `src/main/java/com/ticksense/activities/feedback/OpportunityFeedbackEmitter.java`:

```java
public final class OpportunityFeedbackEmitter
{
    public void accept(OpportunityMarker marker, ExecutionFeedbackSink sink)
}
```

The emitter should check:

- `marker.getStatus() == OpportunityStatus.COMPLETED`
- context contains a zero-loss signal, such as `tickLoss=0`, `latencyTicks=0`, or a tracker-specific field explicitly documented by the tracker.
- evidence includes confirming evidence.

If existing markers do not include standardized latency/tick-loss context, add a minimal helper or context key convention:

```java
public final class FeedbackContextKeys
{
    public static final String TICK_LOSS = "tickLoss";
    public static final String LATENCY_TICKS = "latencyTicks";
}
```

Do not guess from marker time alone.

#### Before / After
**Before:**
```java
opportunitySink.accept(marker);
```

**After:**
```java
opportunitySink.accept(marker);
opportunityFeedbackEmitter.accept(marker, feedbackSink);
```

#### Migration Notes
Existing opportunity markers remain valid. Feedback emits only when context is explicit enough.

#### Misinterpretations To Prevent
- Wrong interpretation: completed means perfect.
  Correct rule: only zero-loss completed opportunities produce `PERFECT_TICK`.
- Wrong interpretation: emit feedback at opportunity open.
  Correct rule: emit only after terminal completion evidence.
- Wrong interpretation: use this for likely misclicks.
  Correct rule: this task only emits positive feedback.

#### Invariants
- `Tick Perfect!` only appears for `PERFECT_TICK`.
- No feedback is emitted for `FAILED`, `EXPIRED`, or `CANCELLED`.
- No feedback is emitted without explicit zero-loss context.
- Config master toggle must be available to suppress downstream presentation, but domain tests can still verify event generation separately.

#### Acceptance Criteria
- [ ] A mapper emits `PERFECT_TICK` feedback for a confirmed zero-loss completed opportunity.
- [ ] Tests prove non-perfect terminal states do not emit perfect feedback.
- [ ] Tests prove completed-but-late opportunities do not emit perfect feedback.
- [ ] `.\gradlew.bat test` passes.

#### Testing Guidance
