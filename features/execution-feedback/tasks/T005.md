### T005 - Conservative Likely Misclick Detection

#### Objective
Detect likely misclicks only after the player corrects them and TickSense observes confirmation evidence. Start with prayer and inventory recovery patterns where intent can be inferred conservatively.

#### Scope
- Add a likely-misclick classifier under `com.ticksense.activities.feedback`.
- Use click position from `PlayerActionTelemetryEvent`.
- Correlate a near-target click followed by a successful intended action within a short tick window.
- Emit `ExecutionFeedbackType.LIKELY_MISCLICK` or `RECOVERED` domain feedback.
- Add synthetic replay/unit tests for prayer-adjacent and food-adjacent correction patterns.

#### Out of Scope
- Do not classify every missed click.
- Do not support NPC/object misclicks in the first version.
- Do not render text or play sound.
- Do not tell the player to click again.
- Do not infer intent from coordinates alone.

#### Blocked By
T001, T002, T003

#### Why It Is Blocked
Likely misclicks require the feedback model, click coordinates, and config sensitivity.

#### Implementation Details
Create `LikelyMisclickClassifier`:

```java
public final class LikelyMisclickClassifier
{
    public Optional<ExecutionFeedbackEvent> observe(ActivityId activityId, TelemetryEvent event, ActivityContext context)
}
```

Start with inventory item correction because inventory consumption confirmation already exists:

- First click has valid canvas point.
- First click is near an inventory slot or target family but does not produce consumption.
- Next relevant click within `1` to `3` ticks consumes food or changes potion dose.
- No unrelated action occurs between first and correction click.
- Emit `LIKELY_MISCLICK` with label `Likely misclick`.

Prayer support should be gated:

- If normalized prayer state telemetry does not exist yet, add a blocker note and only support prayer through debug/fixture tests after T005 has confirmed evidence.
- Do not fake prayer activation from widget click alone.

Evidence detail example:

```text
Likely food misclick: first click was near inventory slot 12, corrected 1 tick later by Eat Shark.
```

#### Before / After
**Before:**
```java
// Food tracker can complete recovery, but cannot explain a nearby failed click before the correction.
```

**After:**
```java
Optional<ExecutionFeedbackEvent> feedback = classifier.observe(activityId, telemetryEvent, context);
```

#### Migration Notes
No existing reports or telemetry should break. If click position fields are unknown, classifier returns empty.

#### Misinterpretations To Prevent
- Wrong interpretation: near click equals misclick.
  Correct rule: near click plus correction plus confirmation evidence is required.
- Wrong interpretation: label as definite misclick.
  Correct rule: always say `Likely misclick`.
- Wrong interpretation: show feedback before correction.
  Correct rule: emit only after successful correction confirmation.
- Wrong interpretation: implement prayer without prayer state evidence.
  Correct rule: prayer misclicks require prayer activation confirmation or remain fixture-blocked.

#### Invariants
- No feedback is emitted for unknown click coordinates.
- No feedback is emitted without a correction action.
- No feedback is emitted without confirmation evidence.
- Default sensitivity is conservative.
- Misclick confidence must be below `1.0D`.

#### Acceptance Criteria
- [ ] Food/inventory likely misclick unit test emits `LIKELY_MISCLICK` after correction.
- [ ] Near-click-without-correction test emits no feedback.
- [ ] Correction outside the configured tick window emits no feedback.
- [ ] Prayer misclick is either implemented with confirmation evidence or explicitly blocked in tests/docs.
- [ ] `.\gradlew.bat test` passes.

#### Testing Guidance
Add `LikelyMisclickClassifierTest`. Use synthetic `PlayerActionTelemetryEvent` with canvas coordinates and `InventoryDeltaTelemetryEvent` for consumption. Add fixture notes for prayer if not implemented.

#### Agent Handoff Context
This feature is valuable because it explains why a player lost a tick, but false positives would feel awful. Bias hard toward conservative detection.

## Final Notes

- Recommended implementation order: after click telemetry is replayable.
- Unresolved questions: inventory slot bounds and prayer widget bounds source.
- Risks: coordinate-based inference can be brittle on resizable layouts.
- Areas that need human review before implementation: sensitivity thresholds and wording.
