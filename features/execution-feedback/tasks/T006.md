### T006 - Feedback Rendering And Sound

#### Objective
Render configured after-the-fact feedback text and play the optional tiny perfect sound from feedback events, without strategies or trackers knowing about UI.

#### Scope
- Add a RuneLite/UI feedback presenter.
- Add a lightweight text rendering path for `Tick Perfect!` and correction labels.
- Add optional sound only for perfect feedback at first.
- Respect master, text, sound, placement, and cooldown config.
- Add focused presenter tests where possible and manual smoke instructions.

#### Out of Scope
- Do not add new feedback types.
- Do not infer misclicks.
- Do not play sounds for likely misclicks by default.
- Do not create future-action overlays or countdowns.

#### Blocked By
T003, T004

#### Why It Is Blocked
Rendering needs config and at least one feedback event source.

- `src/main/java/com/ticksense/runelite/TickSensePlugin.java` - plugin lifecycle and injection.
- `src/main/java/com/ticksense/ui/TickSensePanel.java` - existing UI entry point.
- `src/main/java/com/ticksense/runelite/TickSenseConfig.java` - feedback toggles.

#### Implementation Touchpoints
- Read first: `TickSensePlugin.startUp`, `TickSensePlugin.shutDown`, `TickSenseConfig`, RuneLite overlay examples in the pinned dependency or RuneLite source.
- Edit first: create a presenter under `src/main/java/com/ticksense/ui/feedback/`.

- Required structural changes: add `ExecutionFeedbackPresenter`, `ExecutionFeedbackTextOverlay`, and optional `ExecutionFeedbackSoundPlayer`.
- Do not edit: feedback classifier rules or telemetry schema.
- Verify in order: unit tests for cooldown/config gating, then `.\gradlew.bat test`, then manual `.\gradlew.bat run`.
- Stop and ask: before adding bundled audio assets or choosing a nonstandard audio API.

#### Implementation Details
Create `ExecutionFeedbackPresenter`:

```java
public final class ExecutionFeedbackPresenter implements ExecutionFeedbackSink
{
    public void accept(ExecutionFeedbackEvent feedback)
}
```

Presenter responsibilities:

- If `config.executionFeedback()` is false, suppress all presentation.
- If feedback type is `PERFECT_TICK` and `perfectTickText()` is true, enqueue text.
- If feedback type is `PERFECT_TICK` and `perfectTickSound()` is true, play the tiny sound.
- If feedback type is `LIKELY_MISCLICK` and `correctionFeedbackText()` is true, enqueue text.
- Apply `feedbackCooldownTicks()` per feedback type.

Possible overlay shape:

```java
public final class ExecutionFeedbackTextOverlay extends Overlay
{
    public Dimension render(Graphics2D graphics)
}
```

The overlay must render only after events are received. It must not inspect game state to tell the player what to do next.

#### Migration Notes
No report or telemetry migration.

#### Misinterpretations To Prevent
- Wrong interpretation: overlay can watch live mechanics.
  Correct rule: overlay only displays already-emitted feedback events.
- Wrong interpretation: play sound for mistakes by default.
  Correct rule: first sound path is perfect-only and separately configurable.
- Wrong interpretation: use strategy classes to draw.
  Correct rule: strategies never render.

#### Invariants
- Master toggle suppresses visual and audio feedback.
- Sound is optional and off unless configured.
- No forward-looking text is rendered.
- Feedback expires quickly and is rate-limited.

#### Acceptance Criteria
- [ ] Presenter respects all relevant config toggles.
- [ ] Perfect feedback text can be displayed from a synthetic event.
- [ ] Optional sound path is gated by `perfectTickSound()`.
- [ ] Manual smoke notes confirm no text appears before an action outcome.
- [ ] `.\gradlew.bat test` passes.

#### Testing Guidance
Add presenter unit tests using a fake config and fake sound player. Manual QA with `.\gradlew.bat run` should verify placement and cooldown.

#### Agent Handoff Context
