### T007 - Reports And Replay Coverage

#### Objective
Persist feedback evidence into reports and replay fixtures so Perfect Tick and likely-misclick behavior can be audited after the session.

#### Scope
- Add feedback entries to activity report data or report metadata.
- Include feedback in report text where useful.
- Add replay fixtures for zero-loss perfect feedback and likely food/inventory misclick.
- Add golden or focused assertions for feedback entries.
- Add debug replay coverage for feedback events if debug recorder support is added.

#### Out of Scope
- Do not add new immediate rendering.
- Do not tune classifier thresholds.
- Do not add AI/network summaries.
- Do not enable boss-specific feedback without verified replay evidence.

#### Blocked By
T004, T005

#### Why It Is Blocked
Reports need real feedback events from both positive and correction lanes before report assertions are meaningful.

#### Related Code
- `src/main/java/com/ticksense/analytics/ActivityReport.java` - report model.
- `src/main/java/com/ticksense/analytics/ReportBuilder.java` - report construction interface.
- `src/main/java/com/ticksense/ui/ReportTextFormatter.java` - player-facing report text.
- `src/main/java/com/ticksense/storage/JsonReportRepository.java` - report persistence.
- `src/test/java/com/ticksense/replay/ReplayRegressionTest.java` - replay/golden tests.
- `src/test/resources/replays/` and `src/test/resources/golden/` - fixture and golden output.

#### Implementation Touchpoints
- Read first: `ActivityReport`, `ActivityReportAssembler`, `ReportTextFormatter`, `ReplayRegressionTest`.
- Edit first: add report-level feedback collection field or metric list.
- Symbols to search: `ActivityReportData`, `OpportunityTimelineEntry`, `GoldenReportAssert`, `DebugEventKind`.
- Required structural changes: add report feedback entry model with type, label, time, confidence, and evidence summary.
- Do not edit: classifier rules except to expose events to report assembly.
- Verify in order: focused report tests, replay tests, then full tests.
- Stop and ask: if adding feedback to report JSON requires schema versioning beyond current report compatibility.

#### Implementation Details
Create a report-friendly model, either in analytics or activities:

```java
public final class ExecutionFeedbackSummary
{
    private final String type;
    private final String label;
    private final int gameTick;
    private final double confidence;
    private final List<String> evidenceSummary;
}
```

Add a field to `ActivityReport`:

```java
private final List<ExecutionFeedbackSummary> feedback;
```

Use empty list as default for old reports if deserialization requires it.

Report text examples:

```text
Tick-perfect moments: 4
Likely misclicks: 1
Tick 412: likely food misclick, corrected 1 tick later
```

Replay fixtures:

- `src/test/resources/replays/execution-feedback-perfect-basic.jsonl`
- `src/test/resources/replays/execution-feedback-likely-food-misclick.jsonl`

Golden fixtures may be added under `src/test/resources/golden/` if the report builder path is stable.

#### Before / After
**Before:**
```java
ActivityReport report = builder.build(session, data, opportunityMarkers);
```

**After:**
```java
ActivityReport report = builder.build(session, data, opportunityMarkers, feedbackEvents);
```

If changing `ReportBuilder` is too broad, add feedback via `ActivityReportAssembler` instead and keep builder compatibility where possible.

#### Migration Notes
Existing reports should show no feedback entries rather than failing. Use empty-list defaults if JSON deserialization touches report feedback.

#### Misinterpretations To Prevent
- Wrong interpretation: immediate feedback is enough.
  Correct rule: reports must include feedback evidence so the player can review it later.
- Wrong interpretation: all likely misclicks are certain report facts.
  Correct rule: report text must say `likely`.
- Wrong interpretation: add AI summary.
  Correct rule: no AI/network work in this feature.

#### Invariants
- Feedback report entries are local-only.
- Feedback evidence is visible enough to audit.
- Perfect and likely-misclick counts are separate.
- Reports do not show live instructions.

#### Acceptance Criteria
- [ ] Reports can include zero or more feedback summaries.
- [ ] Existing reports/tests remain compatible.
- [ ] Replay or golden tests prove perfect feedback appears in report output.
- [ ] Replay or golden tests prove likely misclick appears as `likely`.
- [ ] `.\gradlew.bat test` passes.

#### Testing Guidance
Add focused report model tests plus replay regression coverage. Update `ReportTextFormatterTest` to assert player-facing text.

#### Agent Handoff Context
This task makes feedback useful beyond the pop-up. It should explain "why did I lose that tick?" in the final report.

## Final Notes

- Recommended implementation order: after event generation and likely misclick detection.
- Unresolved questions: whether feedback belongs in `ActivityReportData` or final `ActivityReport`.
- Risks: report schema churn and golden fixture noise.
- Areas that need human review before implementation: exact report wording.
