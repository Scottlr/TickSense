### T003 - RuneLite Event Capture Surface

#### Goal
Subscribe to MVP RuneLite events and capture their ordering context without mapping them into the core telemetry model yet.

#### Scope
- Add event subscriber methods for MVP event coverage in `TickSensePlugin` or a delegated `RuneLiteEventCapture`.
- Track current game tick, client tick sequence, client cycle, wall time, monotonic time, FPS, world, and game state.
- Create `RuneLiteClock` and `RuneLiteEventEnvelope`.
- Capture only safe references needed for later mapping; do not store live RuneLite objects beyond the event call.

#### Out of scope
- No normalized telemetry classes, strategy activation, storage, report generation, or debug file writing.
- Do not use `MapRegionChanged`.

#### Migration Notes
- No persisted data is created.
- The event envelope is an adapter-internal type and may change in T006 if needed.

#### Misinterpretations To Prevent
- Wrong interpretation: capture can hold `NPC`, `Player`, or `Widget` fields for later use.
  Correct rule: snapshot inside the event call or leave mapping to T006; do not retain live RuneLite objects.
- Wrong interpretation: event capture should infer gem mining or Araxxor.
  Correct rule: T003 captures event context only.

#### Invariants
- All event subscribers are observe-only.
- Menu entries are not consumed, swapped, injected, reordered, or mutated.
- Region changes are inferred from world-view/game-state/location evidence, not `MapRegionChanged`.
