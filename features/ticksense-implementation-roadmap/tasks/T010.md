### T010 - JSONL Timeline Repository

#### Goal
Persist and read replayable normalized timelines and activity/opportunity markers using local JSONL-first storage.

#### Scope
- Create `TimelineRepository` interface.
- Implement `JsonlTimelineRepository`.
- Add `TimelineRecord`, `TimelineRecordType`, and persisted record support for activity and opportunity markers.
- Write `schema-version.json` and timeline files under `~/.runelite/ticksense/timelines/`.
- Add corrupt-line and schema-version handling.

#### Out of scope
- No non-JSONL database, binary analytics format, sync feature, or external data path.
- No report repository; T017 owns report persistence.
- No retention/trends; T022 owns post-MVP maintenance.
- No `ConfigManager` persistence for timelines or markers.

#### Implementation Notes
- Required interface:

```java
public interface TimelineRepository extends AutoCloseable
{
    void append(TelemetryEnvelope event) throws IOException;
    void appendActivityMarker(ActivityMarker marker) throws IOException;
    void appendOpportunityMarker(OpportunityMarker marker) throws IOException;
    CompletedActivityTimeline readActivityTimeline(ActivityId activityId) throws IOException;
    List<TimelineRecord> readActivityRecords(ActivityId activityId) throws IOException;
    @Override
    void close() throws IOException;
}
```

- `TimelineRecord` should distinguish at least `TELEMETRY_EVENT`, `ACTIVITY_MARKER`, and `OPPORTUNITY_MARKER`.
- `CompletedActivityTimeline` fields: `ActivityId activityId`, `List<TelemetryEnvelope> telemetryEvents`, `List<ActivityMarker> activityMarkers`, `List<OpportunityMarker> opportunityMarkers`.
- Corrupt lines should be skipped with a warning counter returned or exposed for tests, not silently ignored.
- File names should include date, session ID, and `.jsonl`.
- `TickSenseDataPaths` should resolve a plugin-owned directory under `RuneLite.RUNELITE_DIR` at the RuneLite edge and pass plain `Path` values into core storage.
- Do not perform append/read/repair work on the client thread; write tests around repository classes and keep caller wiring responsible for scheduling.

#### Migration Notes
- Schema version starts at `1`.
- Future schema changes must either preserve read compatibility or fail with a clear unsupported-version error.

#### Misinterpretations To Prevent
- Wrong interpretation: add a database because querying is easier.
  Correct rule: JSONL-first storage is required for MVP.
- Wrong interpretation: put timeline JSON in `ConfigManager` because other plugins store JSON there.
  Correct rule: `ConfigManager` is acceptable for settings/small profile data only; TickSense timelines and markers are file-backed.
- Wrong interpretation: repository can compute activity metrics during append.
  Correct rule: timeline storage is append/read only and does not interpret gameplay.

#### Invariants
- JSONL lines are independently parseable.
- Every persisted event has schema version and session ID.
- Storage remains local under TickSense-owned paths.
- No storage task adds SQLite, H2, Parquet, native/JNI, or new JSON/storage dependencies.

#### Acceptance Criteria
- [ ] Appending telemetry creates a JSONL timeline file under the architecture-approved path.
- [ ] Timeline files are written under a TickSense-owned subdirectory of `RuneLite.RUNELITE_DIR`/`~/.runelite`.
- [ ] No timeline or marker data is stored through `ConfigManager`.
- [ ] Reading an activity timeline returns telemetry plus activity/opportunity markers for that activity marker range.
- [ ] Reports can be built from persisted opportunity markers without relying on in-memory tracker state.
- [ ] Corrupt-line handling is tested and does not crash the whole read.
- [ ] Schema version handling is tested.
- [ ] `./gradlew test` passes.
