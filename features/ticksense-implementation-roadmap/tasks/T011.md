### T011 - Activity Strategy Engine Lifecycle

#### Goal
Implement strategy candidate arbitration, active activity lifecycle, termination handling, marker emission, and lifecycle diagnostics.

#### Scope
- Create `ActivityStrategyEngine`.
- Evaluate all registered strategies for each telemetry event.
- Start the highest-confidence valid activity according to architecture tie-breakers.
- Suppress normal activity/report creation when no strategy crosses threshold or when competing evidence remains ambiguous.
- Maintain one primary active activity unless nested spans are explicitly created by the strategy.
- Emit activity markers to a sink for storage.
- Emit lifecycle diagnostics only when debug diagnostics are enabled.

#### Out of scope
- No gem mining, Araxxor, or other activity-specific logic.
- No opportunity terminal-state implementation beyond using the `OpportunitySink` from T008.
- No UI display of diagnostics; T021 owns developer diagnostics UI.
- No normal-user manual select/start/stop/switch behavior.

#### Implementation Notes
- Tie-breaker order:
  1. Existing active strategy persists unless strong termination evidence appears.
  2. Highest confidence above threshold.
  3. More specific strategy beats generic strategy.
  4. Boss/raid/minigame strategy beats generic skilling while in combat instance.
  5. Suppressed candidates are diagnostic-only.
- If no candidate reaches threshold, the timeline remains generic telemetry only and no normal report is generated.
- If two candidates cannot be separated confidently, suppress normal activity creation and emit debug diagnostics only when enabled.
- Suggested constructor:

```java
public final class ActivityStrategyEngine implements TelemetrySink
{
    public ActivityStrategyEngine(ActivityRegistry registry,
                                  ActivityMarkerSink activityMarkerSink,
                                  OpportunitySink opportunitySink,
                                  boolean diagnosticsEnabled);

    @Override
    public void accept(TelemetryEnvelope envelope);
}
```

#### Migration Notes
- Activity marker format should align with T008 `ActivityMarker`.
- Engine diagnostics are local debug artifacts and should not be persisted as normal reports.

#### Misinterpretations To Prevent
- Wrong interpretation: engine should contain boss/skilling heuristics.
  Correct rule: engine arbitrates; strategy classes own mechanics.
- Wrong interpretation: low-confidence candidates should become normal reports.
  Correct rule: low-confidence candidates remain diagnostics only.
- Wrong interpretation: ambiguous detection should ask the player to choose the current activity.
  Correct rule: ambiguous detection suppresses normal reports and never prompts the user to select, start, stop, or switch activity.

#### Invariants
- Engine never imports RuneLite or Swing.
- Every finished active activity has a `FinishReason`.
- Candidate suppression is deterministic and explainable in diagnostics.
- No-confidence and ambiguous evidence paths produce no normal completed report.

#### Acceptance Criteria
- [ ] Engine starts an activity when candidate confidence reaches threshold.
- [ ] Existing active activity persists until termination evidence appears.
- [ ] Engine emits start and finish activity markers.
- [ ] Engine suppresses lower-priority candidates and records diagnostic evidence when enabled.
- [ ] Engine creates no normal report for no-confidence or ambiguous detection.
- [ ] Engine exposes no normal-user manual activity selection, start, stop, or switch path.
- [ ] `./gradlew test` passes.
