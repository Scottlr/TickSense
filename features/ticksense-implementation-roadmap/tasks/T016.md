### T016 - Metrics Engine and Timing Calculators

#### Goal
Add reusable metric models and timing calculators for ticks, milliseconds, seconds, percentiles, downtime, and explainable score breakdowns.

#### Scope
- Create `MetricDefinition`, `MetricValue`, `MetricUnit`, `MetricSeries`, `TimingCalculator`, `PercentileCalculator`, `ExecutionScore`, and `ScoreBreakdown`.
- Calculate recognition/action/execution latency, idle ticks, downtime, p50/p90/p95, best/worst, and score components.
- Keep scoring secondary to raw evidence.

#### Out of scope
- No activity-specific formulas beyond generic helpers.
- No UI rendering.
- No trend analysis across reports; T022 owns trends.

#### Migration Notes
- Metric keys should be stable because reports and trend storage will reference them.
- If a formula changes later, update schema/version notes and golden reports.

#### Misinterpretations To Prevent
- Wrong interpretation: always convert everything to ticks.
  Correct rule: use ticks for server-cycle execution, milliseconds for client-observed reactions, and seconds for downtime.
- Wrong interpretation: score is the main report.
  Correct rule: score is secondary and must be explainable by raw evidence.

#### Invariants
- Do not report ping unless a supported API is verified.
- Tick loss is attributed only when expected timing is known.
- Unattributed downtime remains visible.
