### T017 - Report Model and Repository

#### Goal
Define stable report models and persist completed report summaries and full report JSON locally.

#### Scope
- Create `ActivityReport`, `ReportSummary`, `OpportunityTimelineEntry`, `TickLossBreakdown`, and `ReportRepository`.
- Implement `JsonReportRepository`.
- Save reports under `~/.runelite/ticksense/reports/`.
- Maintain `~/.runelite/ticksense/indexes/report-index.json`.
- Add delete-all-data storage control.

#### Out of scope
- No Swing rendering; T018 owns UI.
- No trends/retention beyond basic listRecent; T022 owns post-MVP maintenance.
- No separate summary/export models beyond local report JSON.
- No `ConfigManager` persistence for reports or report indexes.

#### Implementation Notes
- Required repository interface:

```java
public interface ReportRepository
{
    void save(ActivityReport report) throws IOException;
    Optional<ActivityReport> findById(String reportId) throws IOException;
    List<ReportSummary> listRecent(int limit) throws IOException;
}
```

- `ActivityReport` fields should include `schemaVersion`, `reportId`, `activityId`, `activityType`, `detectedActivityName`, `createdAtMillis`, `durationTicks`, `durationMillis`, `finishReason`, `confidence`, `evidenceSummary`, `metrics`, `opportunities`, `tickLossBreakdown`, and `summaryLines`.
- `ReportSummary` should include `detectedActivityName`, `finishReason`, `confidence`, and enough evidence summary text for the detail view to explain why the report exists without showing raw telemetry.
- Delete-all-data must remove TickSense-owned files only under the configured TickSense data root.
- Report writes should use a temporary file plus atomic/replace move where Java 11 and the filesystem allow it; index rebuild must tolerate missing, corrupt, or partial report files.

#### Before / After
Before:

```text
Analytics can create metric values, but no stable report object exists.
```

After:

```java
reportRepository.save(activityReport);
List<ReportSummary> recent = reportRepository.listRecent(20);
```

#### Migration Notes
- Report JSON starts with schema version `1`.
- Add `schemaVersion` to both summary index and full report files.
- Future schema migrations should rebuild the index from full reports.

#### Misinterpretations To Prevent
- Wrong interpretation: UI panel owns report persistence.
  Correct rule: repository owns persistence; UI reads summaries and report details.
- Wrong interpretation: store recent reports in `ConfigManager` so they follow RuneLite profiles.
  Correct rule: reports and indexes are local file data under TickSense-owned storage; `ConfigManager` remains for settings only.
- Wrong interpretation: delete-all-data can delete parent directories broadly.
  Correct rule: delete only resolved paths under `~/.runelite/ticksense/`.

#### Invariants
- Reports are generated from completed activities only.
- Completed reports record detected activity name, finish reason, confidence, and evidence summary.
- Stored reports contain no raw chat, player names, or account identifiers.
- Report repository has no RuneLite dependency.
- Reports and indexes are file-backed, not `ConfigManager`-backed.

#### Acceptance Criteria
- [ ] Reports save and load round-trip with schema version.
- [ ] Reports persist detected activity name, finish reason, confidence, and evidence summary.
- [ ] Reports and report index are written as files under TickSense-owned storage, not `ConfigManager`.
- [ ] `listRecent(limit)` reads from report index.
- [ ] Index rebuild handles a missing or corrupt index.
- [ ] Delete-all-data removes only TickSense-owned data root contents.
- [ ] `./gradlew test` passes.
