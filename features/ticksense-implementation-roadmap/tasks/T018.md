### T018 - RuneLite Report Panel and Detail Views

#### Goal
Display completed reports in a RuneLite side panel with player-facing language and no live gameplay guidance.

#### Scope
- Fill `TickSensePanel` with recent report list, report detail view, opportunity timeline, and tick-loss breakdown.
- Add refresh after a report is saved.
- Show low-confidence reports only in debug/developer diagnostics, not in normal recent reports.
- Show completed reports only; do not include any control or prompt asking what activity the user is doing.
- Add settings entry points for delete-all-data and debug toggles if config already exists.
- Hide or disable the Trends entry until T022 implements trend summaries.

#### Out of scope
- No live overlays, live status instructions, sounds, target suggestions, or mechanic prompts.
- No normal-user manual activity select/start/stop/switch UI.
- No trend charts; T022 owns trends.
- No non-local report sections.

#### Implementation Notes
- Normal panel structure:

```text
TickSense
Recent Reports
Trends
Settings
Developer Diagnostics (debug only)
```

- Detail panel labels should include "Duration", "Grade", "Total tick loss", "Best execution", "Worst execution", and "Opportunity timeline" where data exists.
- Detail panel should show the detected activity name, finish reason, confidence, and short evidence summary from the report model.
- Avoid front-facing words: telemetry, inference, profiling, pipeline.
- If a Trends label is present in MVP, it must be disabled with no content panel until T022.

#### Migration Notes
- UI should tolerate older/missing report fields by displaying "Unknown" or hiding optional sections, not crashing.
- Report repository schema remains authoritative.

#### Misinterpretations To Prevent
- Wrong interpretation: show active strategy or current opportunity in normal UI.
  Correct rule: normal UI shows completed reports only.
- Wrong interpretation: add "what activity are you doing?" selection to resolve ambiguous detection.
  Correct rule: ambiguous detection is suppressed by the engine; normal UI never asks the user to choose or switch activity.
- Wrong interpretation: terms like telemetry are acceptable because players are technical.
  Correct rule: use OSRS-friendly report language from the architecture.

#### Invariants
- UI has no real-time mechanic instructions.
- UI has no normal-user manual activity selection, start, stop, or switch controls.
- UI depends on report models and repositories, not strategy internals.
- Low-confidence reports do not appear as normal completed reports.
